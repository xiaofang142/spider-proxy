    if (bytesRead > 0) {

private suspend fun forwardData() {
    while (isConnected) {
        val packet = DatagramPacket(buffer, buffer.size)
        socket?.receive(packet)

        if (packet.length > 0) {
            val data = packet.data.copyOf(packet.length)

            // 将响应写回 TUN 设备
            tunDeviceWriter?.writeUdpResponse(
                udpData = data,
                srcAddress = destAddress,
                srcPort = destPort,
                destAddress = srcAddress,
                destPort = srcPort
            ) ?: run {
                Log.d(TAG, "UDP received ${packet.length} bytes")
            }
        }
    }
}

## 四、完整流量处理流程

### 4.1 出站流量（设备 → 代理）

```
1. TUN 设备读取 IP 数据包
   ↓
2. VpnService.forwardTraffic() 解析协议
   ↓
3. 根据协议选择转发器
   - TCP (6) → TcpForwarder.handleTcpPacket()
   - UDP (17) → UdpForwarder.handleUdpPacket()
   ↓
4. 转发器管理连接并发送数据到代理服务器
```

### 4.2 入站流量（代理 → 设备）

```
1. 代理服务器返回响应
   ↓
2. TcpForwarder/UdpForwarder 读取响应
   ↓
3. 调用 TunDeviceWriter.writeTcpResponse()/writeUdpResponse()
   ↓
4. TunDeviceWriter.buildIpPacket() 构建完整 IP 包
   - 添加 IP 头部（源地址=目标服务器，目标地址=VPN 地址 10.0.0.1）
   - 计算校验和
   ↓
5. 写入 TUN 设备
   ↓
6. 系统接收数据包并路由到应用
```

## 五、设计决策

### 5.1 可选依赖

`TunDeviceWriter` 作为可选依赖（默认值为 `null`），原因：

- **向后兼容** - 现有代码无需修改即可使用
- **测试友好** - 可以独立测试转发器逻辑
- **灵活部署** - 支持不同场景的配置

### 5.2 地址映射

写回 TUN 设备时的地址映射：

```
代理响应:
  源：目标服务器 (例如 1.2.3.4:80)
  目标：客户端 (例如 10.0.0.2:12345)

写回 TUN:
  源：目标服务器 (1.2.3.4:80)  ← IP 包源地址
  目标：VPN 客户端 (10.0.0.1:12345)  ← IP 包目标地址
```

### 5.3 简化实现

当前实现的简化之处：

- **TCP 序列号/确认号** - 未修改，直接使用原始数据
- **NAT 转换** - 未实现完整的 NAT 表
- **SOCKS5 UDP ASSOCIATE** - 使用直接转发模式

这些简化在大多数场景下可以正常工作，但可能在复杂网络环境下需要完善。
